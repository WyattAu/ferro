## Running Benchmarks

```bash
cargo bench -p ferro-server
```

To run a specific benchmark group:

```bash
cargo bench -p ferro-server -- upload_throughput
cargo bench -p ferro-server -- request_latency
cargo bench -p ferro-server -- webdav_ops
cargo bench -p ferro-server -- wasm_dispatch
cargo bench -p ferro-server -- "storage/in_memory"
cargo bench -p ferro-server -- "metadata/in_memory"
```

## Benchmark Suites

### Upload Throughput (`throughput.rs`)

Measures upload performance for sequential and concurrent file uploads.

| Benchmark | Description |
|---|---|
| `upload_throughput/sequential/1KB` | Single-threaded 1KB upload |
| `upload_throughput/sequential/1MB` | Single-threaded 1MB upload |
| `upload_throughput/sequential/10MB` | Single-threaded 10MB upload |
| `upload_throughput/concurrent_Nclients/1KB` | N concurrent 1KB uploads (N=10,50,100) |
| `upload_throughput/concurrent_Nclients/1MB` | N concurrent 1MB uploads (N=10,50,100) |

### Download Throughput (`throughput.rs`)

Measures download performance including full response body transfer.

| Benchmark | Description |
|---|---|
| `download_throughput/sequential/1KB` | Single-threaded 1KB download |
| `download_throughput/sequential/1MB` | Single-threaded 1MB download |
| `download_throughput/sequential/10MB` | Single-threaded 10MB download |
| `download_throughput/concurrent_Nclients/1KB` | N concurrent 1KB downloads (N=10,50,100) |
| `download_throughput/concurrent_Nclients/1MB` | N concurrent 1MB downloads (N=10,50,100) |

### Request Latency (`latency.rs`)

Measures per-request latency for GET, PUT, and DELETE at various payload sizes.

| Benchmark | Description |
|---|---|
| `request_latency/put/1KB` | PUT latency with 1KB payload |
| `request_latency/put/1MB` | PUT latency with 1MB payload |
| `request_latency/put/10MB` | PUT latency with 10MB payload |
| `request_latency/get/1KB` | GET latency for 1KB file |
| `request_latency/get/1MB` | GET latency for 1MB file |
| `request_latency/get/10MB` | GET latency for 10MB file |
| `request_latency/delete/1KB` | DELETE latency for 1KB file |
| `request_latency/delete/1MB` | DELETE latency for 1MB file |
| `request_latency/delete/10MB` | DELETE latency for 10MB file |

### WebDAV Operations (`webdav_ops.rs`)

Measures latency of WebDAV-specific operations.

| Benchmark | Description |
|---|---|
| `webdav_ops/mkcol` | Collection creation (MKCOL) |
| `webdav_ops/propfind/10` | PROPFIND listing 10 items |
| `webdav_ops/propfind/100` | PROPFIND listing 100 items |
| `webdav_ops/propfind/1000` | PROPFIND listing 1000 items |
| `webdav_ops/propfind/depth0` | PROPFIND depth:0 (single item metadata) |
| `webdav_ops/delete/file` | Delete a single file |
| `webdav_ops/delete/recursive/10` | Delete collection with 10 children |
| `webdav_ops/delete/recursive/100` | Delete collection with 100 children |

### WASM Dispatch (`wasm_dispatch.rs`)

Measures WASM worker infrastructure overhead.

| Benchmark | Description |
|---|---|
| `wasm_dispatch/runtime_creation` | Wasmtime engine instantiation |
| `wasm_dispatch/register_worker` | Worker registration time |
| `wasm_dispatch/find_matching/1` | Pattern matching with 1 registered worker |
| `wasm_dispatch/find_matching/10` | Pattern matching with 10 registered workers |
| `wasm_dispatch/find_matching/50` | Pattern matching with 50 registered workers |
| `wasm_dispatch/find_matching/100` | Pattern matching with 100 registered workers |
| `wasm_dispatch/dispatch_overhead/noop_module` | Full dispatch path (nonexistent module) |

## Comparison with Nextcloud

To compare Ferro against Nextcloud:

1. **Run Ferro benchmarks**:
   ```bash
   cargo bench -p ferro-server -- --save-baseline ferro
   ```

2. **Run Nextcloud benchmarks** using the same payload sizes (1KB, 1MB, 10MB)
   via a tool like [wrk](https://github.com/wg/wrk) or [hey](https://github.com/rakyll/hey):
   ```bash
   hey -n 10000 -c 10 http://nextcloud-host/remote.php/dav/files/admin/test.txt
   ```

3. **Compare results**:
   ```bash
   cargo bench -p ferro-server -- --baseline ferro
   ```

Key differences to account for:
- Ferro benchmarks use in-memory storage (no disk I/O); Nextcloud uses PHP+MySQL
- Ferro is a single Rust binary; Nextcloud requires Apache/Nginx + PHP-FPM + MySQL
- Ferro benchmarks include full HTTP stack (axum + middleware); Nextcloud benchmarks
  include the full LAMP stack overhead
